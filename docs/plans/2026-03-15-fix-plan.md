# Mission Control — Fix Plan (2026-03-15)

## Context

The transcript describes a hyper-personalized Mission Control dashboard built inside OpenClaw. The plan specified that the app should ingest **real** OpenClaw filesystem state as its source of truth — not use placeholder or hardcoded data. The current implementation has the skeleton screens built, but almost all data is hardcoded or the ingestion layer points to wrong paths and never actually pulls real data in.

---

## Gap Analysis

### 1. Ingestion Layer — Wrong Paths / No Real Data

**Problem:** `src/lib/openclaw/ingestion.ts` uses hardcoded paths but:
- Looks for `task-*.md` files in `/Volumes/Data/openclaw/workspace/tmp/` — that folder has task files like `task-008`, `task-009`, etc. ✅ (exists)
- Ingests projects from `/Volumes/Data/openclaw/workspace/projects/` (real projects exist here) ✅
- Does **NOT** ingest memories from `/Volumes/Data/openclaw/workspace/memory/` ❌
- Does **NOT** ingest docs from workspace `/Volumes/Data/openclaw/workspace/docs/` ❌
- Does **NOT** ingest schedule jobs from any real cron/schedule source ❌
- Does **NOT** ingest activity log ❌

**Fix:** Update `ingestion.ts` to:
- Read `memory/YYYY-MM-DD.md` files → populate `memories` table
- Read `tmp/task-*.md` files as tasks → already partially done, needs path fix
- Read `docs/*.md` files and `memory/reference/` → populate `documents` table
- Seed some representative schedule jobs from `launchd/` or heartbeat config

---

### 2. Task Board — No Real Tasks Show

**Problem:** The DB is never seeded on first run. The Ingest button exists but clicking it only works if the `ingestWorkspace()` runs successfully. The `async` wrapping in `ingestion.ts` uses `db.transaction(async () => ...)` but `better-sqlite3` only supports **synchronous** transactions — this is a **runtime bug** silently failing.

**Fix:**
- Convert all DB operations in `ingestion.ts` to synchronous (remove async/await wrappers inside transactions)
- Also auto-run ingestion on first load if DB is empty
- Add an `ensureSeeded()` call at app startup in `layout.tsx`

---

### 3. Memories Screen — Shows No Real Memories

**Problem:** The `getMemories()` function queries the DB, but memories are never ingested — the DB table is always empty. The UI falls back to the hardcoded example.

**Fix:**
- Ingest all `memory/YYYY-MM-DD.md` files through `ingestion.ts`
- Group by date, show most recent at top
- The left sidebar should show real date-labeled entries

---

### 4. Docs Screen — Shows No Real Docs

**Problem:** Same as memories — `documents` table always empty. Docs screen shows hardcoded example content.

**Fix:**
- Ingest markdown files from `workspace/tmp/`, `workspace/memory/`, and `workspace/docs/` into the `documents` table
- Each file becomes a `DocumentEntry` with title = filename, path = absolute path, category = inferred from folder

---

### 5. Calendar — No Real Schedule Jobs

**Problem:** `schedule_jobs` table always empty. Calendar falls back to no slots.

**Fix:**
- Seed representative schedule jobs from launchd plist files or heartbeat config
- At minimum, parse `/Volumes/Data/openclaw/workspace/launchd/` for `.plist` files

---

### 6. Projects — Wrong Progress & Metadata

**Problem:** All projects get `progress: 65` hardcoded. No task count, no actual linkage.

**Fix:**
- After ingesting tasks, compute progress as `completedTasks / totalTasks * 100`
- Link tasks to projects by matching project name in task metadata

---

### 7. Team Screen — Mission Statement is Generic Placeholder

**Problem:** The mission statement shown on the Team page is a hardcoded generic string. The real mission statement lives in `SOUL.md` or `MEMORY.md`.

**Fix:**
- Read the mission statement from `SOUL.md` at ingestion time
- Store it as a special agent note or site-level config in DB

---

### 8. Office Screen — Agent Status Always Idle

**Problem:** All agents always show as idle/online. There is no live status.

**Fix (pragmatic):** Read which tasks are currently "In Progress" or "Implementation" and map them to the relevant agent, showing that agent as "busy." This is achievable without real-time infra.
